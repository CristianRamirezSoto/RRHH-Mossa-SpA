LEVANTAMIENTO DEL SISTEMA RRHH MOSSASPA
=======================================

Objetivo
--------
Ejecutar la app de RRHH con React/Vite + Firebase, primero en ambiente local
con emuladores y luego conectada a un proyecto real de Firebase.


1. Requisitos instalados
------------------------
- Node.js 20 o superior.
- npm.
- Firebase CLI:
  npm install -g firebase-tools


2. Instalación inicial
----------------------
Desde la carpeta raíz del proyecto:

  npm install
  cd functions
  npm install
  cd ..


3. Credenciales del frontend
----------------------------
El archivo real de credenciales locales es:

  .env.local

Ese archivo NO se debe subir a GitHub ni compartir públicamente.

Para crearlo:

  1. Copiar .env.example como .env.local.
  2. Entrar a Firebase Console.
  3. Ir a Configuración del proyecto > Tus apps > SDK setup and configuration.
  4. Copiar los valores web en las variables VITE_FIREBASE_*.

Variables necesarias:

  VITE_FIREBASE_API_KEY
  VITE_FIREBASE_AUTH_DOMAIN
  VITE_FIREBASE_PROJECT_ID
  VITE_FIREBASE_STORAGE_BUCKET
  VITE_FIREBASE_MESSAGING_SENDER_ID
  VITE_FIREBASE_APP_ID
  VITE_FIREBASE_VAPID_KEY

Nota: estas credenciales web identifican el proyecto Firebase, pero la seguridad
real debe quedar en Firebase Auth, Firestore Rules, Storage Rules y Cloud Functions.


4. Uso local con emuladores
---------------------------
Para trabajar sin gastar lecturas/escrituras reales de Firebase:

  1. En .env.local dejar:

     VITE_USE_EMULATORS=true

  2. Levantar emuladores:

     npm run emulators

  3. En otra terminal levantar la app:

     npm run dev

  4. Abrir la URL que entrega Vite, normalmente:

     http://localhost:5173

Panel de emuladores:

  http://localhost:4000


5. Uso conectado a Firebase real
--------------------------------
Para probar contra Firebase real:

  1. En .env.local dejar:

     VITE_USE_EMULATORS=false

  2. Verificar que .firebaserc apunte al proyecto correcto:

     firebase use --add

     Elegir el proyecto real y asignarlo como default.

  3. Iniciar sesión:

     firebase login

  4. Levantar app:

     npm run dev


6. Deploy formal
----------------
Antes de publicar:

  npm run build

Luego desplegar reglas, funciones y hosting:

  firebase deploy

Si se quiere separar:

  firebase deploy --only firestore:rules,storage
  firebase deploy --only functions
  firebase deploy --only hosting


7. Primer usuario administrador
-------------------------------
La función createUserProfile crea automáticamente el primer usuario como admin.

Proceso recomendado:

  1. Crear el primer usuario desde la app.
  2. Ese usuario queda como administrador inicial.
  3. Los siguientes usuarios quedan como employee por defecto.


8. Módulos Firebase usados
--------------------------
- Authentication: login de usuarios.
- Firestore: empleados, documentos, perfiles, asistencia, notificaciones.
- Storage: archivos del expediente digital.
- Cloud Functions: operaciones privilegiadas como biometría, asistencia y alertas.
- Hosting: publicación web.
- Cloud Messaging: notificaciones push si se configura VAPID.


9. Archivos que no se deben versionar
-------------------------------------
- .env
- .env.local
- node_modules/
- functions/node_modules/
- dist/
- .firebase/
- firebase-debug.log
- firestore-debug.log
- ui-debug.log
- emulator-data/
- *.out.log
- *.err.log


10. Flujo recomendado de pruebas
--------------------------------
1. Probar primero con emuladores.
2. Crear un admin.
3. Crear trabajadores.
4. Registrar consentimiento biométrico.
5. Enrolar rostro.
6. Probar marcaje de entrada.
7. Probar marcaje de salida.
8. Revisar el módulo Asistencia.
9. Probar expediente digital.
10. Recién después probar Firebase real.


11. Recomendación técnica
-------------------------
Para este proyecto conviene seguir con Firebase porque la app ya usa Auth,
Firestore, Storage, Functions, Hosting, Messaging y emuladores. Migrar ahora a
otra base de datos agregaría costo de tiempo sin una ganancia clara inmediata.

Supabase sería buena alternativa si el sistema necesitara SQL relacional fuerte,
reportes complejos, joins frecuentes o control PostgreSQL directo. Para esta
fase, Firebase es más rápido y consistente con lo ya construido.
